Hata Bildirme

PHP güvenliğiyle hata bildirmenin iki yönü vardır. Birinde güvenliği arttırıcı yararlar vardır. Diğerinde ise zarar vericidir.

A standard attack tactic involves profiling a system by feeding it improper data, and checking for the kinds, and contexts, of the errors which are returned. This allows the system cracker to probe for information about the server, to determine possible weaknesses. For example, if an attacker had gleaned information about a page based on a prior form submission, they may attempt to override variables, or modify them:

Kişiselleştirilmiş HTML sayfalarıyla değişkenlerin ilgisini çekmek 
<form method="post" action="attacktarget?username=badfoo&password=badfoo">
<input type="hidden" name="username" value="badfoo" />
<input type="hidden" name="password" value="badfoo" />
</form>
The PHP errors which are normally returned can be quite helpful to a developer who is trying to debug a script, indicating such things as the function or file that failed, the PHP file it failed in, and the line number which the failure occurred in. This is all information that can be exploited. It is not uncommon for a php developer to use show_source(), highlight_string(), or highlight_file() as a debugging measure, but in a live site, this can expose hidden variables, unchecked syntax, and other dangerous information. Especially dangerous is running code from known sources with built-in debugging handlers, or using common debugging techniques. If the attacker can determine what general technique you are using, they may try to brute-force a page, by sending various common debugging strings:

Hata düzeltme değişkenlerini açma 
<form method="post" action="attacktarget?errors=Y&showerrors=1&debug=1">
<input type="hidden" name="errors" value="Y" />
<input type="hidden" name="showerrors" value="1" />
<input type="hidden" name="debug" value="1" />
</form>
Hatayı ele alma yöntemine bakmaksızın, hatalar için sistemi irdeleme yeteneği daha çok bilginin ilgisini çekmesini sağlar.

Örneğin soysal PHP hatasının stili PHP çalıştıran bir sistemi belirtir. Eğer ilgilenen .html sayfasından bakıyorsa, ve arka tarafta olanları irdeleyecekse (sistemin bilinen zayıflıklarına bakmak isterse), onu yanlış bilgiyle besleyerek sistemin PHP ile çalıştığını öğrenmiş olur.

Bir işlev hatası sistemin belirtilen veri tabanı makinesini çalıştırıp çalıştırmadığını belirtebilir. Ya da web sayfasının, programlamanın, tasarımın nasıl olduğunun ipucunu verir. Bu açık veri tabanı port'larına daha derin bir inceleme gerektirir. Değişik parçalara kötü veri besleyerek, örneğin ilgilenen kodlamadaki yetki erişimini belirleyebilir (hata kodlarının satır numaralarından) kodlamanın değişik yerlerine konan açıklayıcı bilgileri irdeleyerek de sağlar.

Bir dosya düzeni ya da PHP hatası web sunucusunun ne özellikleri olduğunu belirtir. Web sunucunda dosyaların yapısı ve düzenlemesi kadar herşey öğrenilebilir. Eğer hata diğer kullanıcıların anlamayacağı biçimde gizlenebilirse geliştirici tarafından yazılan hata kodu bu problemi kapatabilir.

Bu ortamda üç ana çözüm vardır. İlki tüm işlevlere dikkate bakmak, toplu hata üremesini engellemektir. İkincisi hata yazdırmayı tümüyle kaldırmaktır. Ücüncüsü PHP içinde kişiselleştirilmiş hata işlevi ile kendi hata mesajlarınızı oluşturmaktır.

Zamanından önce bu bilgiye erişmek için PHP tarafından sunulan error_reporting() işlevini kullanmaktır. Kendi kodunuzu yayımlamadan önce E_ALL ile denetleyerek kodunuzun nerelerde gösteilmeye ya da değişikliğe uğrayabileceğini bulmaya çalışabilirsiniz. yayımdan önce ya error_reporting() 0 yapılır ya da error_diplay kapatılabilir. Sonrakini kullanacak olursanız, log dosyası için error_log ini yönelticisiyle bir yol belirtebilirsiniz ve log_errors değerini açabilirsiniz.

E_ALL ile tehlikeli değişkenleri bulma 
<?php
if ($username) {
    // Kullanmadan önce ilklememiş ya da denetlenmemiş
    $good_login = 1;
}
if ($good_login == 1) {
    // Eğer deneme hatalıysa, kullanılmadan önce
    // ilklenmemiş ya da denetlenmemiş
    readfile ("/highly/sensitive/data/index.html");
}
?>
